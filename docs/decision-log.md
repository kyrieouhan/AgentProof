# 决策日志

采用简化 ADR。状态为 `accepted` 的决策在被新 ADR 替代前有效。

## ADR-001：先 CLI 后 Web

- 日期：2026-07-11
- 状态：accepted
- 决策：M1 先建立安全 CLI 内核；完整 Web 界面不得先于价值闭环。
- 原因：先证明运行、隔离和证据可信性，降低界面掩盖核心问题的风险。

## ADR-002：先确定性验证，后引入大模型

- 日期：2026-07-11
- 状态：accepted
- 决策：状态码、JSON、DOM、数据库、退出码、哈希和 Diff 优先；模型仅参与结构化建议、计划建议和失败摘要。
- 后果：模型判断不能成为最终验收证据。

## ADR-003：Node.js / TypeScript 优先

- 日期：2026-07-11
- 状态：accepted
- 决策：本地 MVP 只优先支持规范 Node.js / TypeScript Web 项目。
- 后果：Python 等多语言生态属于后续路线。

## ADR-004：M3 是本地 MVP，M4 才加入 GitHub App

- 日期：2026-07-11
- 状态：accepted
- 决策：M3 完成本地浏览器验证与轻量防作弊闭环；M4 才进行 Alpha 发布、外部项目验证和 GitHub App 工作流。

## ADR-005：控制面与执行面分离

- 日期：2026-07-11
- 状态：accepted
- 决策：可信控制面创建和销毁临时执行环境；不可信仓库只在执行面运行。
- 后果：策略、验收规则、Manifest 生成和密钥不得交给被验收代码控制。

## ADR-006：被验收容器不挂载 Docker Socket

- 日期：2026-07-11
- 状态：accepted
- 决策：Docker Engine 只由可信控制面访问；目标容器绝不获得 Docker Socket。
- 原因：Socket 通常等价于对 Docker 主机的高权限控制。

## ADR-007：Zod 是领域 Schema 单一事实来源

- 日期：2026-07-11
- 状态：accepted
- 决策：从 M2 起以 Zod 定义领域 Schema，并生成 JSON Schema。
- 后果：不得手工维护相互漂移的双重领域 Schema。M0 的 failure-case Schema 是独立数据集规范，不等于运行时领域实现。

## ADR-008：证据 Manifest 绑定运行上下文与哈希

- 日期：2026-07-11
- 状态：accepted
- 决策：Manifest 至少绑定 Commit、规则版本、环境、镜像摘要、命令、随机种子和证据文件哈希，并由可信控制面产生。
- 限制：哈希只能证明内容未变；可信性还依赖 Manifest 可信根、密钥和访问控制。

## ADR-009：前期不做用户采访

- 日期：2026-07-11
- 状态：accepted
- 决策：M0 是问题与技术可行性验证，不要求采访、用户意愿或付费验证。
- 后果：不得虚构采访记录、市场反馈或付费数据。

## ADR-010：真实用户与外部项目验证在 M4

- 日期：2026-07-11
- 状态：accepted
- 决策：M4 才邀请真实用户，并在非官方项目上进行 Alpha 验证。
- 后果：M4 的前置依赖不再要求 M3 已完成外部项目验证；外部项目验证属于 M4 工作和退出门禁。

## ADR-011：Failure Case 规范以版本化 Schema 与人工批准为准

- 日期：2026-07-11
- 状态：accepted
- 决策：M0-01 冻结 failure-case 分类、JSON Schema、跨字段语义校验和审核清单版本 `1.0.0`；来源核实、复现和最终审核使用独立状态维度。
- 后果：后续案例必须通过同版本 Schema；只有人工可以设置最终 `approved`，模型候选、合成示例和未通过门禁的记录不能计入 M0 的 20 个案例。
- 变更规则：先更新 Schema、语义校验器、测试夹具、分类和审核文档，再迁移已有记录；不得在单个案例中创建私有字段或同义状态。

## ADR-012：M0 接受 19 条核心候选并继续官方 Demo

- 日期：2026-07-12
- 状态：accepted
- 决策：M0 原计划核心案例目标为不少于 20 条；严格范围审核后，如实保留 19 条核心候选、8 条次级研究和 12 条排除案例，不再为凑数补充第 20 条低质量案例。
- 原因：第 20 条若降低来源真实性、AI Agent 交付关联性或本地 MVP 可检测性标准，会削弱失败基准的可信度。当前 19 条已经覆盖首版主要失败模式，足以支持官方 Demo、Runner 输入和验证器设计。
- 后果：该决定是明确的 M0 门禁例外，不得把统计伪造成 20 条，也不得把未人工批准的候选迁移到正式 `cases/`。如果后续自然发现新的高质量案例，可以补充；但它不再阻断 M0-04 官方 Demo 正确基线工作。
- 边界：本决策不代表 M0 退出门禁全部通过，不允许进入 M1；本轮不设置 `approved`、`source_verified` 或 `reproduced`。

## ADR-013：M0 以已接受数量门禁例外结束并进入 M1

- 日期：2026-07-12
- 状态：accepted
- 决策：M0-08 退出审计确认 M0 可以在 ADR-012 的 19 条核心候选例外下结束，并进入 M1 安全执行内核与 CLI POC。
- 原因：M0 已交付官方 Demo 正确基线、五类缺陷样例、检测证据矩阵、最小可行性实验、Runner 输入规范、安全评审和支持范围评审；第 20 条候选缺口不再阻断 M1 的安全 Runner 开发。
- 后果：M1 不重新扩展 failure-case 搜索，不迁移正式 `cases/`，也不把候选标记为 `approved`、`source_verified` 或 `reproduced`。M1 必须遵守 M0 固定的支持矩阵、威胁模型和 Runner 输入规范。
- 边界：本决策只允许进入 M1；不允许提前开发 M2/M3 能力、GitHub App、完整 Web 或大模型验收判断。

## ADR-014：M1 退出审计通过并进入 M2

- 日期：2026-07-13
- 状态：accepted
- 决策：M1 退出审计确认安全执行内核与 CLI POC 可以结束，并进入 M2 API-first 垂直验收闭环。
- 原因：M1 已完成 RunnerProfile/CLI 契约、Docker 预检、容器生命周期、默认隔离策略、install/build/test 结构化记录、timeout/cancel 清理诊断、三样例 repeatability 和 ISO-01 至 ISO-14 隔离烟测；量化门禁均有本地证据。
- 后果：M2 可以在不削弱 M1 隔离边界的前提下实现领域 Schema、验收项版本、API/数据库断言、Evidence Manifest、脱敏和报告聚合。
- 边界：本决策不允许开发 GitHub App、云端 Runner、完整 Web、浏览器验证器或大模型最终判断；浏览器验证属于 M3，GitHub 工作流属于 M4。

## ADR-015：M2 退出审计通过并进入 M3

- 日期：2026-07-13
- 状态：accepted
- 决策：M2 退出审计确认 API-first 垂直验收闭环可以结束，并进入 M3 浏览器验证与防作弊轻量版。
- 原因：M2 已完成领域 Schema、验收项版本、API/数据库断言、随机数据与 seed、Evidence Manifest、脱敏、报告聚合和官方 Demo 正确/五类缺陷回归；量化门禁均有本地证据。
- 后果：M3 可以在不削弱 M1/M2 边界的前提下增加一条关键 Playwright 浏览器流程、页面/API/数据库联合断言、测试 Diff 风险模型和轻量硬编码检测。
- 边界：本决策不允许开发 GitHub App、云端 Runner、完整 Web、外部 Alpha 或真实用户验证；这些属于 M4 或更晚阶段。

## ADR-016：M3 本地 MVP 完成并停止在 M4 之前

- 日期：2026-07-13
- 状态：accepted
- 决策：M3 退出审计确认浏览器验证与防作弊轻量版本地 MVP 可以结束；当前停止在 M3 完成状态，不自动进入 M4。
- 原因：M3 已完成浏览器步骤 Schema、单流程 Playwright 执行、浏览器证据采集与脱敏、页面/API/SQLite 联合断言、测试/配置 Diff 风险、只读规则、随机化硬编码检测、官方缺陷回归和 3 次一致性。
- 后果：仓库具备本地 MVP 证据链，但 GitHub App、Alpha 发布、外部项目验证、真实用户工作流和完整 Web 仍未实现。
- 边界：进入 M4 必须获得用户明确授权；不得因为 M3 通过而自动创建 GitHub App、配置远程、推送、发布或邀请外部用户。

## ADR-017：M3 补齐最小本地 Web 用户入口

- 日期：2026-07-13
- 状态：Accepted
- 决策：在不进入 M4 的前提下，为 M3 增加最小本地 Web 用户入口；该入口复用现有 CLI、Runner、M3 browser smoke、Evidence Manifest 和报告模块，不重新实现第二套验收引擎。
- 原因：M3-06 的 `m3:regression` 和 `usability=true` 只能证明 CLI/回归/证据链可运行，不能等同于用户可操作的 VeriCrate Web 界面。
- 后果：用户现在可以通过 `npm run web:dev` 打开 `http://127.0.0.1:4173`，导入官方 Demo、编辑/确认验收项、发起真实验收、查看阶段进度和证据，并导出 HTML/Markdown 报告。
- 边界：该入口是本地单机最小界面，不包含账号系统、GitHub App、Alpha 发布、云端部署、外部项目验证或完整多项目平台；这些仍需 M4 或后续明确授权。
