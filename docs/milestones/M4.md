# M4：Alpha 发布与 GitHub 工作流

## 目标

把 M3 的“本地内部可用”转为“陌生用户可安装、可理解、可反馈”，在真实非官方项目上开展 Alpha 验证，并接入最小权限 GitHub App 与 PR Check。

## 前置依赖

M3 全部门禁通过，官方 Demo 正确/缺陷版本稳定，本地安装文档和支持矩阵已准备。外部项目稳定运行不是 M4 的前置条件，而是 M4 的工作与退出门禁。

## 负责人

产品/开发共同负责 Alpha；GitHub 集成负责人实现 App；安全/隐私评审单独审核权限、Webhook、密钥、证据链接和反馈材料。

## 输入

M3 本地 MVP、安装诊断、官方 Demo、支持矩阵、威胁模型、隐私说明草案、Alpha 招募与授权流程、GitHub App 权限设计。

## 工作内容

- 发布清晰标注限制的开源 Alpha 和安装诊断。
- 邀请真实用户，获得明确同意后在非官方项目上运行；记录兼容性、误判、漏判、不稳定和复现材料。
- 不收集或宣称未真实发生的反馈、意愿或付费数据。
- 实现最小权限 GitHub App；PAT 仅作本地调试兜底，不作为正式路径。
- PR 打开/更新触发验证，Check 绑定准确 Head Commit；处理取消、重跑、Webhook 重放和并发更新。
- 实施密钥轮换、证据链接访问控制、反馈脱敏与保留策略。
- 根据真实使用数据决定是否提议 Post-MVP，不能自动扩大范围。

## 交付物

开源 Alpha、安装/诊断文档、GitHub App、PR Check、反馈与授权流程、首批真实用户/外部项目报告、更新后的支持矩阵、威胁模型和隐私说明。

## 明确非目标

不承诺任何仓库都能验收；不提供企业级 SLA、云端多租户、云端 Runner、自动修复、Mutation Testing、多语言或多 Agent 对比。

## 量化退出门禁

- 邀请不少于 10 名真实用户，并保留同意、参与和反馈状态；不得把邀请数等同活跃数。
- 至少 5 个非官方项目完成一次端到端验收，并记录支持条件、结果状态和可复现材料。
- PR 打开或更新后可由 GitHub App 触发验证，Check 准确绑定 Head Commit；过期运行不能覆盖新 Commit。
- 安装失败、识别失败、误判、漏判和 `unstable` 均可匿名反馈并附脱敏复现材料。
- GitHub 权限、Webhook 重放、密钥轮换、证据链接鉴权和数据删除流程通过安全评审。
- 对外文档不含虚假截图、性能、兼容性、市场或付费声明。

## 安全要求

最小 GitHub App 权限；安装私钥/Token 不进入被验收容器；Webhook 验签与重放保护；Check 绑定 Head SHA；报告链接鉴权和短期有效；反馈默认脱敏；真实仓库需授权和数据保留/删除机制。

## 风险

真实项目包含秘密或个人数据、权限过宽、Webhook 竞态、Check 绑定错误 Commit、兼容性远低于预期、反馈样本偏差。关键安全问题覆盖功能完成度并阻止 Alpha 扩大。

## 推荐任务拆分

1. Alpha 安装/诊断、授权、隐私和反馈流程。
2. 小批真实用户/外部项目验证并逐步扩展。
3. GitHub App 最小权限、Webhook 与 Head SHA 状态机。
4. 证据链接鉴权、密钥轮换、取消/重跑。
5. 支持矩阵和威胁模型基于真实结果更新。
6. M4 退出评审与 Post-MVP 立项建议；只建议，不自动实施。

## 里程碑边界

- M3 提供本地 MVP；M4 承接全部真实用户、外部项目 Alpha 与 GitHub 工作流。
- Post-MVP 的 Mutation Testing、多生态、团队版、多 Agent 和云端 Runner 必须根据真实使用数据另行立项。
