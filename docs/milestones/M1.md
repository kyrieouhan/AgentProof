# M1：安全执行内核与 CLI POC

## 目标

在不依赖大模型和 Web 界面的情况下，安全、稳定、可复现地运行支持矩阵内的规范 Node.js 项目，并输出结构化运行记录。安全隔离是 M1 核心能力，不是后置功能。

## 前置依赖

M0 全部门禁通过：官方 Demo/案例/检测矩阵可用，威胁模型和支持矩阵获评审，Runner 输入规范固定。

## 负责人

基础设施/Runner 开发主责；安全评审参与设计、烟测和退出门禁；样例维护者提供 3 个规范仓库。

## 输入

本地 Git 路径、固定 Commit、RunnerProfile、安装/构建/测试命令、Node/包管理器版本、端口、资源/网络策略和临时非敏感配置。

## 工作内容

- 实现最小 CLI 与可信控制面，创建和销毁临时 Linux 容器。
- 识别 `package.json`、lockfile、Node 版本和标准脚本；识别失败允许显式配置。
- 固定镜像 digest、命令、环境版本、随机种子和缓存策略。
- 记录安装/构建/测试的退出码、耗时、stdout/stderr 和结构化失败原因。
- 实施非 root、capability、只读/临时文件系统、网络、CPU、内存、PID、磁盘和超时限制。
- 执行 [threat-model.md](../threat-model.md) 的 M1 隔离烟测。

## 交付物

CLI POC、RunnerProfile Schema、结构化运行记录、容器生命周期与清理、隔离策略、诊断日志、烟测证据和重放说明。

## 明确非目标

不做 GitHub App、自然语言需求拆解、Fastify API、完整 Web、浏览器自动化、Mutation Testing、多语言或云端 Runner。

## 量化退出门禁

- 至少 3 个支持矩阵内样例仓库在全新临时环境完成 install/build/test；失败日志完整可回看。
- 同一 Commit 在固定镜像与种子下连续运行 10 次，基础结果一致且可重放；不一致必须诊断，不能以成功率替代一致性。
- ISO-01 至 ISO-14 隔离烟测全部通过并保存证据。
- 目标容器无法读取宿主个人目录、SSH Key、真实 `.env`、私人凭据或 Docker Socket。
- 超时、取消、失败和清理路径都有自动测试；没有静默遗留容器或敏感临时目录。

## 安全要求

控制面/执行面分离；绝不挂载 Docker Socket；默认拒绝出站；最小权限；资源/时长/日志配额；规则与最终证据不可由目标代码写；Docker 非绝对安全的剩余风险写入报告。

## 风险

依赖安装脚本、Docker Desktop 差异、镜像/registry 漂移、Windows 路径映射、清理失败和日志洪泛。环境失败必须使用 `infrastructure_error`，不能混为 `failed`。

## M1-01 完成结果

- 新增根 CLI：`vericrate profile validate --profile <path> [--repo-root <path>] [--json]`。
- 新增 RunnerProfile JSON Schema：`schemas/runner-profile.schema.json`。
- 新增标准库校验器：`src/runner-profile.mjs`，检查 M1 必需字段、路径边界、Node/package manager、lockfile、端口、健康检查、资源、网络、挂载和证据策略。
- 新增官方 Demo profile：`samples/demo-web-app/vericrate.runner-profile.json`。
- 新增最小测试：`tests/runner-profile.test.mjs`。

M1-01 不启动容器、不访问 Docker、不运行目标仓库命令，也不产生可信 Manifest；这些属于 M1 后续任务。

## M1-02 完成结果

新增 Docker 前置诊断命令：`vericrate docker check [--json]`。该命令只检查 Docker CLI 与 Docker Engine 是否可用，并把缺失运行时报告为 `infrastructure_error`，不把环境问题混同为项目失败。

新增最小生命周期 smoke：`vericrate run --profile <path> --lifecycle-smoke [--json]`。它会把目标项目复制到 `.tmp/vericrate-runs/` 下的临时工作区，拉取或使用 RunnerProfile 指定镜像，以 `--network none`、`--user 1000:1000`、`--cap-drop ALL`、`--security-opt no-new-privileges`、`--pids-limit 64`、`--memory 256m`、`--read-only`、只读 `/workspace` 挂载和 `/tmp` tmpfs 启动容器，并验证 `package.json` 可见、Docker Socket/`.env`/`.npmrc` 不可见。

当前机器实测：`npm run docker:check` 通过，Docker Desktop CLI 位于 `<DOCKER_CLI_PATH>`；`npm run smoke:docker` 通过，拉取 `node:20-alpine`，记录 digest `sha256:fb4cd12c85ee03686f6af5362a0b0d56d50c58a04632e6c0fb8363f609372293`，容器输出 `vericrate-lifecycle-smoke-ok`，临时目录清理结果为 `removed`。

M1-02 只证明容器生命周期和最小挂载可用；更完整的资源/网络/权限策略与 ISO-01 至 ISO-14 隔离烟测属于 M1-03 和 M1-07。

## M1-03 完成结果

新增可复用容器策略模块 `src/runner-policy.mjs`，把 M1-02 内联参数固化为默认策略：`--network none`、`--user 1000:1000`、`--cap-drop ALL`、`--security-opt no-new-privileges`、`--pids-limit 64`、`--memory 256m`、`--cpus 1`、`--read-only`、只读 `/workspace` 和 `/tmp` tmpfs。

`npm run smoke:docker` 已真实验证：容器用户为非 root；Docker Socket 不存在；`.env` 与 `.npmrc` 未被复制；`/workspace` 和根文件系统不可写；向 `1.1.1.1:80` 的 Node TCP 连接未成功；临时目录清理为 `removed`。

M1-03 不等于 ISO-01 至 ISO-14 全量隔离烟测通过；它只把首批资源、网络、权限和挂载策略变成可复用代码并进行最小真实验证。

## M1-04 完成结果

新增结构化命令运行入口：`vericrate run --profile <path> --commands [--json]`，根脚本为 `npm run run:demo`。Runner 会把目标项目复制到 `.tmp/vericrate-runs/` 临时工作区，检查或拉取 RunnerProfile 指定镜像，然后按 `install`、`build`、`test` 三个阶段分别启动受限容器，并在同一个临时 workspace 中保留安装产物。

运行记录现在包含每条 Docker 命令的 `phase`、完整命令、退出码、stdout、stderr、开始/结束时间和耗时。命令容器使用 RunnerProfile 的 CPU、内存、PID 和 `command_timeout_ms` 限制；官方 Demo profile 已调整为 `node:20-bookworm`、`pnpm@10.17.1`、`pids=512` 和 `command_timeout_ms=600000`，构建阶段显式运行 Prisma Client 生成。

当前机器实测：`npm run run:demo` 通过，runId 为 `m1-smoke-mriskluq`，镜像 digest 为 `sha256:8f693eaa7e0a8e71560c9a82b55fd54c2ae920a2ba5d2cde28bac7d1c01c9ba5`。`install` 阶段使用 `--network bridge`，退出码 0，耗时 283336ms；`build` 阶段使用 `--network none`，退出码 0，耗时 39725ms；`test` 阶段使用 `--network none`，退出码 0，耗时 23989ms，官方 Demo 测试结果为 8 passed。临时目录清理结果为 `removed`。

M1-04 只完成 install/build/test 的结构化运行记录。更细的依赖源白名单、取消/超时诊断、3 个样例、10 次重复运行和 ISO-01 至 ISO-14 全量隔离烟测仍属于后续 M1 任务。

## M1-05 完成结果

Runner 现在会在每条 Docker 命令记录中保留 `errorCode`、`signal`、`timedOut` 和 `cancelled` 诊断字段。install/build/test 与 lifecycle smoke 会优先把 timeout 和 cancelled 分类为顶层 `timeout` / `cancelled` 状态，而不是笼统归为 `failed`。

当容器命令 timeout 或 cancelled 时，Runner 会对已命名容器执行 `docker rm -f <container>`，并把该清理命令记录到对应命令的 `containerCleanup` 字段。临时 workspace 删除失败时不会吞掉原始运行结果，而是在结果中保留 `cleanup="failed"` 和 `cleanupError`。

自动测试新增 timeout 与 cancelled 路径覆盖：伪造 run layer 返回 `ETIMEDOUT` 或 `cancelled=true` 时，Runner 必须返回对应状态、保留错误原因、执行容器强制清理，并删除临时 workspace。真实 Docker timeout smoke 也已通过：临时 profile 把 install 命令设为 10 秒睡眠、`command_timeout_ms=1000`，Runner 返回 `timeout`，记录 `docker rm -f` 清理，且 `docker ps -a` 确认无遗留容器。

M1-05 不新增交互式 UI 取消按钮，也不完成 3 个样例、10 次重复运行或 ISO-01 至 ISO-14 全量烟测；这些仍属于后续 M1 任务。

## M1-06 完成结果

新增 3 个支持矩阵内轻量 npm 样例用于 Runner repeatability：`samples/minimal-npm-api/`、`samples/minimal-npm-state/` 和 `samples/minimal-npm-files/`。三个样例均为 Node.js 20、npm lockfile、显式 install/build/test/start 命令、HTTP `/health`、无外部依赖、无真实 `.env` 或私人凭据。

新增 `npm run m1:repeatability`，由 `scripts/run-m1-repeatability.mjs` 顺序运行三个样例的 RunnerProfile，并对 `minimal-npm-state` 连续运行 10 次。脚本生成 `artifacts/m1-repeatability/summary.json` 和 `artifacts/m1-repeatability/report.md`，记录每次 runId、镜像 digest、phase 退出码、网络策略、清理状态和 repeatability 签名。

当前机器实测：`npm run m1:repeatability` 通过；三个样例均完成 install/build/test，三阶段退出码均为 0，临时 workspace 清理均为 `removed`，镜像 digest 为 `sha256:8f693eaa7e0a8e71560c9a82b55fd54c2ae920a2ba5d2cde28bac7d1c01c9ba5`。`minimal-npm-state` 连续 10 次运行均为 `passed`，唯一签名数为 1。

官方 Demo 仍作为 M1-04 的复杂 pnpm/SQLite/Prisma 样例保留；M1-06 的重复稳定性选择零外部依赖样例，以避免 native dependency 在安装阶段重新拉取 Node headers 时引入外部网络波动。

## M1-07 完成结果

新增 `npm run m1:isolation`，由 `scripts/run-m1-isolation-smoke.mjs` 执行 ISO-01 至 ISO-14 的隔离烟测，并生成 `artifacts/m1-isolation-smoke/summary.json` 与 `artifacts/m1-isolation-smoke/report.md`。

当前机器实测：14 项隔离烟测全部通过，runId 为 `m1-isolation-smoke-mriu27bv`，Docker CLI 为 `<DOCKER_CLI_PATH>`，Docker server 为 `29.6.1`，镜像 digest 为 `sha256:8f693eaa7e0a8e71560c9a82b55fd54c2ae920a2ba5d2cde28bac7d1c01c9ba5`。

烟测覆盖宿主路径不可见、SSH/Git/npm/env 凭据不复制、Docker Socket 不存在、只读 workspace、默认无网络、内存/PID/CPU 限制可观测、文件数配额受控失败、timeout 后容器清理、非 root/no-new-privileges/drop capabilities、路径穿越边界、目标代码不能写可信 Manifest、敏感金丝雀不进入容器环境、repeatability digest 证据可查，以及清理失败注入可见可重试。

Runner 同步补上最小资源配额检查：命令阶段结束后会检查 stdout/stderr 字节数、workspace 文件数和 workspace 字节数；超过 RunnerProfile 限额时返回受控 `failed` 并在命令记录中写入 `resourceLimitExceeded`。临时目录清理也支持注入失败，便于验证清理异常会被记录为 `cleanup="failed"` 和 `cleanupError`。

本结果不宣称 Docker 是绝对安全边界；它只证明 M1 本地 MVP 的默认隔离策略、清理路径和证据记录满足当前威胁模型要求。更强的 VM/远程隔离、签名 Manifest、API 断言和浏览器验证属于后续里程碑或更高安全等级。

## 里程碑边界

- 上一个 M0 提供问题基准、Demo、支持和安全约束；M1 不重新扩展范围。
- 下一个 M2 才引入验收标准、API 验证和证据报告。M1 只证明安全可复现运行。
