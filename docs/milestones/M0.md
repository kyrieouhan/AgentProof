# M0：问题与技术可行性验证

## 定位与目标

M0 不证明市场规模，也不要求用户采访、付费意愿或真实项目 Alpha。它用可追溯失败案例、官方 Demo 和最小实验回答：常见失败能否稳定复现，AgentProof 设想的检测方式能否产生明确证据，以及 M1 应支持什么项目与安全边界。

目标：

- 建立可复现的编程 Agent 失败案例数据集。
- 建立官方 Demo 的正确版本与缺陷版本。
- 验证核心检测思路在技术上可行。
- 明确支持边界、威胁模型和后续 Runner 输入规范。

## 前置依赖

无产品或代码依赖。文档依赖为总计划、统一术语、威胁模型草案、支持矩阵和 failure-case Schema。

## 负责人角色

- 产品/研究负责人：案例来源、分类、去重、隐私与审核主责。
- 测试/验证设计：复现步骤、预期检测方式和证据类型主责。
- Node.js 开发者：官方 Demo 设计与缺陷可复现性支持。
- 安全评审：威胁模型、Runner 输入和 M1 门禁审核。

## 输入

- 原始项目计划与最新范围决策。
- 公开 Issue、事故复盘、已获授权的匿名材料或团队自有记录。
- [failure-case Schema](../../datasets/failure-cases/schema.json)。
- [failure-case 分类体系](../../datasets/failure-cases/category-taxonomy.md)与[人工审核清单](../../datasets/failure-cases/review-checklist.md)。
- [威胁模型](../threat-model.md)与[支持矩阵](../support-matrix.md)。

## 主要任务

1. **案例数据集**：收集、核实、去重、脱敏和分类不少于 20 个失败案例；记录来源类型、来源引用核实状态、复现状态和预期证据。
2. **分类体系**：使用 M0-01 冻结的 `missing_implementation`、`superficial_completion`、`weakened_tests`、`authorization_bypass`、`hardcoded_behavior`、`non_persistent_state`、`ui_api_inconsistency`、`build_runtime_mismatch`、`incomplete_edge_cases`、`false_completion_claim`、`data_integrity_failure`、`security_regression`、`nondeterministic_result`、`environment_dependency`；`needs_review` 只允许临时待审，不得进入批准案例。
3. **官方 Demo 设计**：建立正确版本和缺陷版本，至少稳定复现 5 类缺陷；每类有触发条件、预期行为、实际行为、检测方法和证据类型。
4. **检测可行性实验**：用最小手工或一次性实验验证 API、数据库、Diff、刷新前后状态、权限上下文等证据能否区分正确与缺陷版本。实验结果必须真实记录，不得写成已实现产品能力。
5. **Runner 输入规范**：明确 M1 接受的仓库、Node/包管理器/lockfile、命令、端口、环境变量、资源与网络策略字段。
6. **安全与支持评审**：完成威胁模型和支持矩阵，明确 M1 的保守 Node.js 项目范围与隔离烟测。

## 交付物

- `datasets/failure-cases/` 中冻结的分类、Schema、流程、审核清单和 ≥20 个通过审核且符合计数规则的案例。
- 官方 Demo 正确版本与至少 5 类缺陷版本，以及复现说明。
- 每类缺陷的预期检测方式与证据类型矩阵。
- 完整 `docs/threat-model.md` 与 `docs/support-matrix.md`。
- M1 Runner 输入规范和项目接受清单。
- M0 退出评审记录；未满足项不得标记完成。

## 非目标

- 不要求用户采访、市场反馈、用户意愿或付费验证。
- 不开发正式 Runner、CLI、API、前端、需求解析器、Playwright 验证器或 GitHub App。
- 不安装项目依赖或创建完整 Monorepo。
- 不把模型生成案例冒充真实来源案例。
- 不证明所有业务需求都可自动验收。

### 2026-07-12 核心案例数量门禁例外

M0 原计划要求不少于 20 个核心失败案例。经过严格范围审核后，当前候选池如实记录为 19 条 `core_in_scope`、8 条 `secondary_research`、12 条 `out_of_scope`，正式 `datasets/failure-cases/cases/` 仍为空，未执行 `approved`、`source_verified`、`reproduced` 或正式迁移。

为避免降低来源真实性、AI Agent 交付关联性和本地 MVP 可检测性标准，本项目不再为了凑数补充第 20 条低质量案例。该决定是明确的 M0 门禁例外，不得把统计伪造成 20 条。

19 条核心候选已经覆盖首版主要失败模式，足以支持官方 Demo、Runner 输入和验证器设计。若后续自然发现新的高质量案例，可以补充；但第 20 条不再作为进入后续 M0 工作的阻断条件。M0-08 退出审计已接受该例外，并允许进入 M1。

## 安全要求

案例和 Demo 不得含真实凭据或未授权个人数据。来源材料必须脱敏并遵守授权范围。M1 输入规范必须默认拒绝 Docker Socket、宿主个人目录、SSH Key、真实 `.env` 和无限制网络/资源。

## 风险

- 案例数量达标但同质化：以根因、触发条件和预期证据去重。
- 来源难核实：保留记录但明确 `primary_source.source_verification_status=source_pending` 且 `verified=false`，不得批准或计入 M0 的 20 个正式案例。
- Demo 为检测器量身定做：至少同时保留合理业务上下文、正确版本和多个缺陷路径。
- 可行性实验被误称为产品：报告明确实验方式、限制和未实现部分。

## 必须人工完成

- 判断来源是否合法、是否可引用、是否已核实。
- 审核脱敏是否充分，尤其是截图、日志和业务数据。
- 判定两个案例是否同一根因。
- 运行或确认真实复现，并签署 M0 退出评审。
- 审批安全边界和任何支持范围扩展。

## 可以交给 Codex

- 按 Schema 校验、规范化和统计已提供的案例。
- 发现缺失字段、重复候选和术语不一致，供人工确认。
- 生成 Demo 缺陷设计草案、复现检查表和证据矩阵草案。
- 执行经批准的本地复现步骤并保存真实输出。
- 运行文档、JSON、链接和一致性检查。

Codex 不得编造来源、采访、反馈、复现或测试结果。

## 与 M1 的边界

M0 只证明问题/检测思路并固定输入和安全约束；M1 才实现安全 CLI 执行内核。M0 门禁未通过时，不创建 Runner 业务代码。
