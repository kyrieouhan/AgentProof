# M2：API-first 垂直验收闭环

## 目标

打通最短价值链：自然语言需求 → 用户可编辑验收项 → API/数据库断言 → 可信证据 → 逐项结果与合并建议。

## 前置依赖

M1 全部门禁通过；Runner 稳定；官方 Demo API、数据库种子和固定环境可重放。

## 负责人

全栈/领域开发主责；产品负责人确认验收标准；Runner 与安全负责人维护执行边界和证据可信根。

## 输入

固定 Commit、已确认需求、结构化验收项、RunnerProfile、API 目标、测试数据/种子、数据库检查与证据保留策略。

## 工作内容

- 以 Zod 作为验收标准和运行时领域 Schema 的单一事实来源并生成 JSON Schema。
- 支持用户编辑、确认和版本化验收项；信息不足返回 `insufficient_spec`。
- 实现 API 验证：状态码、JSON 字段/Schema、响应头、错误信息、耗时和数据库前后状态。
- 使用随机化测试数据并保存种子。
- 由可信控制面生成 Manifest，绑定 Commit、规则、镜像、命令、环境、种子和证据哈希。
- 生成基础 HTML/Markdown 报告，并脱敏请求头、Cookie、Token 和个人数据。
- 实现统一结果与合并建议聚合，不用模糊总分。

## 交付物

验收标准 Schema、用户确认/版本机制、API 验证器、基础数据库检查、证据 Manifest、结果聚合、基础 HTML/Markdown 报告和回归测试。

## 明确非目标

不做复杂 Web 项目管理、完整前端、Playwright 浏览器验证、录像、GitHub 自动触发、Mutation Testing、多语言或总分模型。

## 量化退出门禁

- Demo 至少 5 类需求可生成结构化验收项，用户修改后保存为独立版本。
- 每个结论可追溯到必要的请求、响应、数据库状态、Commit、命令和证据哈希。
- 六个底层状态和四个合并建议与 [glossary.md](../glossary.md) 完全一致。
- `infrastructure_error` 不计为功能失败；`unverifiable` 不被忽略；阻断级安全问题覆盖普通完成度。
- 至少一组金丝雀 Token/Cookie/个人数据脱敏测试通过。
- 官方 Demo API 正确/缺陷路径进入自动回归。

## 安全要求

延续 M1 隔离；API 响应和日志视为不可信输入；限制大小、超时和重定向；报告转义；Manifest 在执行面外生成；敏感数据最小采集与分层存储。

## 风险

自然语言歧义、Schema 漂移、随机数据引入不稳定、报告解析注入、脱敏遗漏、数据库断言破坏状态。关键条件不足时应停止并使用 `insufficient_spec` 或 `unverifiable`。

## 推荐任务拆分

1. [x] M2-01 领域模型、状态与 Zod Schema。
2. [x] M2-02 验收项编辑/确认/版本。
3. [x] M2-03 API 与数据库断言。
4. [x] M2-04 随机数据与种子。
5. [x] M2-05 证据采集、Manifest 与脱敏。
6. [x] M2-06 HTML/Markdown 报告与聚合。
7. [x] M2-07 Demo 五类需求端到端回归与安全复审。

## M2-01 完成结果

新增 Zod 领域 Schema 单一事实源：`src/domain/status.mjs` 定义六个底层结果状态和四个合并建议，`src/domain/schemas.mjs` 定义验收项、API 断言、数据库断言、证据引用、验证运行和验证报告。

新增 `npm run schema:generate`，由 `scripts/generate-domain-schemas.mjs` 从 Zod 生成 JSON Schema，并写入 `schemas/domain/`。

状态聚合遵守 [glossary.md](../glossary.md)：`failed` 或阻断级安全问题导向 `do_not_merge`；`infrastructure_error` 或 `insufficient_spec` 导向 `indeterminate`；`unverifiable` 或 `unstable` 导向 `human_review`；全部 `passed` 时才 `recommend_merge`。`status_counts` 明确包含六个底层状态键，不使用模糊总分。

当前校验：`npm run schema:generate`、`npm run lint` 和 `npm test` 通过；测试总数为 21。M2-01 不实现验收项持久化版本、API 执行、数据库检查、Manifest、脱敏或报告渲染，这些属于后续 M2 任务。

## M2-02 完成结果

新增验收项版本模型：`src/domain/acceptance-versions.mjs` 支持创建 draft 版本、基于上一版本生成独立修订版本、显式用户确认版本，以及阻止未确认版本进入验证运行。

`src/domain/schemas.mjs` 新增 `AcceptanceCriterionVersionSchema` 和确认状态 Schema；`npm run schema:generate` 现在额外生成 `schemas/domain/acceptance-criterion-version.schema.json`。版本记录包含 `version_id`、`version_number`、`previous_version_id`、`change_summary`、`created_by`、`created_at` 和 `confirmation`，并要求外层 `criterion_id` 与内层验收项一致。

自动测试覆盖：初始 draft 创建、编辑产生独立 draft v2 且不修改 v1、未确认版本不能进入 run、确认必须记录 `confirmed_by` 与 `confirmed_at`、修订不得改变 `criterion_id`。

当前校验：`npm run schema:generate`、`npm run lint` 和 `npm test` 通过；测试总数为 25。M2-02 不实现 UI、数据库持久化、API 执行、Manifest、脱敏或报告渲染。

## M2-03 完成结果

新增最小断言引擎：`src/domain/assertions.mjs` 支持通过 Node 内置 `fetch` 运行 API 断言，也支持对已采集的 API 响应和数据库快照进行本地判断。

API 断言检查状态码、可选响应头、JSON 字段子集和耗时上限；数据库断言检查 `check_id` 与期望快照字段子集。`evaluateCriterionAssertions` 会把同一验收项下的 API/数据库断言聚合为 CriterionResult 形状，任一断言失败则返回 `failed` 并保留具体错误。

自动测试覆盖：本地 HTTP server 的 API 断言通过路径、状态码/JSON/耗时失败路径、数据库快照字段匹配与失败路径、验收项聚合结果。当前校验：`npm run schema:generate`、`npm run lint` 和 `npm test` 通过；测试总数为 29。

边界：M2-03 不启动官方 Demo 服务、不直接连接 SQLite、不生成 Manifest、不脱敏、不渲染报告；数据库断言当前接收可信控制面后续采集的快照对象，真实数据库连接和证据哈希属于 M2-05 前后的工作。

## M2-04 完成结果

新增可重放测试数据生成器：`src/domain/test-data.mjs` 提供 `createSeededRng`、`demoRegistrationData` 和 `seedRecord`。同一 seed 会生成一致的整数、token 和 Demo 注册数据；不同 seed 会生成不同邮箱和密码。

`src/domain/schemas.mjs` 新增 `TestDataSeedSchema`，`npm run schema:generate` 现在生成 `schemas/domain/test-data-seed.schema.json`。种子记录包含 `schema_version`、`seed`、`purpose` 和 `values`，用于后续 Manifest 与报告追踪测试数据来源。

自动测试覆盖：同 seed 确定性、不同 seed 差异性、Demo 注册数据格式，以及 seed 与生成值同记录保存。当前校验：`npm run schema:generate`、`npm run lint` 和 `npm test` 通过；测试总数为 32。

边界：M2-04 的随机生成器用于可重放测试数据，不是密码学随机数；本次不实现 Manifest、脱敏、报告渲染、数据库持久化或官方 Demo 端到端回归。

## M2-05 完成结果

新增证据与 Manifest 基础：`src/domain/manifest.mjs` 提供 `hashContent`、`evidenceRef` 和 `createEvidenceManifest`，用于把运行上下文、证据路径、证据哈希、seed 和脱敏摘要绑定到可信控制面生成的 Manifest。

新增字段级脱敏：`src/domain/redaction.mjs` 递归替换敏感键名，包括 authorization、cookie、password、token、secret 和 api key 类字段，并返回 `redacted_count` 与 `redacted_paths`。`src/domain/schemas.mjs` 新增 `EvidenceManifestSchema`、`RedactionSummarySchema`，`npm run schema:generate` 现在生成 `schemas/domain/evidence-manifest.schema.json`。

自动测试覆盖：敏感请求头和嵌套 token/password 脱敏、非敏感字段保留、证据内容 sha256 哈希、Manifest 绑定 run_id/commit/runner_profile/image_digest/seed/evidence/redaction。当前校验：`npm run schema:generate`、`npm run lint` 和 `npm test` 通过；测试总数为 34。

边界：M2-05 不做签名密钥、长期证据存储、访问控制、报告渲染或浏览器 Trace；Manifest 当前是本地 MVP 的结构化可信根，不宣称达到发布级证据签名。

## M2-06 完成结果

新增报告与聚合模块：`src/domain/report.mjs` 提供 `createVerificationReport`、`renderMarkdownReport` 和 `renderHtmlReport`。

聚合逻辑继续使用六个底层状态和四个合并建议，不引入总分：`failed` 或阻断级安全问题导向 `do_not_merge`；`infrastructure_error`/`insufficient_spec` 导向 `indeterminate`；`unverifiable`/`unstable` 导向 `human_review`；全部通过时才 `recommend_merge`。

Markdown 报告输出 run、合并建议、状态计数和逐验收项表格；HTML 报告对 run id、summary 和 errors 做 HTML 转义，避免把目标输出当作可执行标记。当前校验：`npm run schema:generate`、`npm run lint` 和 `npm test` 通过；测试总数为 37。

边界：M2-06 不提供完整 Web UI、样式系统、证据文件落盘、签名、访问控制或浏览器 Trace；下一步才把官方 Demo 正确/缺陷路径串成端到端回归。

## M2-07 完成结果

新增官方 Demo 端到端回归脚本：`npm run m2:demo-regression`，由 `scripts/run-m2-demo-regression.mjs` 复用 Demo 正确基线 probe 和五类官方 defect patch verifier，并把结果包装为 M2 report、Manifest、seed 和证据哈希。

当前回归结果写入 `artifacts/m2-demo-regression/`：正确基线通过，五类官方缺陷 `superficial_completion`、`authorization_bypass`、`weakened_tests`、`hardcoded_behavior`、`non_persistent_state` 均被区分；报告包含 6 个 criterion，`status_counts.passed=6`，合并建议为 `recommend_merge`。

为避免 `pnpm exec` 在回归时触发 build-approval 状态检查，Demo 缺陷验证脚本改为调用本地 `node_modules/.bin/tsx`，weakened_tests 探针改为直接执行 `node scripts/test.mjs`。这不改变缺陷语义，只减少包管理器副作用。

安全复审：M2 回归继续使用 M1 的 RunnerProfile、固定镜像 digest 记录、seed、证据 sha256 和字段级脱敏摘要；正式 `datasets/failure-cases/cases/` 仍为空，公开候选没有被标记为 `approved`、`source_verified` 或 `reproduced`。

当前校验：`npm run lint`、`npm run m2:demo-regression`、`npm run schema:generate`、`npm test`、JSON 解析、Markdown 本地链接、failure-case 统计、`cases/` 为空和 `git diff --check` 通过。M2-07 不开发浏览器验证器、GitHub App、完整 Web 或 M3 能力。

## 里程碑边界

- M1 提供可信运行和原始命令记录；M2 不削弱隔离。
- M3 才加入一条关键浏览器流程、测试 Diff 和轻量防作弊；M2 保持 API-first。
