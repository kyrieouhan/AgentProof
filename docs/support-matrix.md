# 本地 MVP 支持矩阵

本矩阵描述目标支持承诺，不表示当前 M0 已实现任何能力。第一版保守定义；未列明的组合默认为“暂不支持”，不得自行推断兼容。

## 支持等级

| 等级 | 含义 |
| --- | --- |
| 支持 | 纳入官方回归、文档和兼容性承诺。 |
| 有条件支持 | 仅在列出的前提下支持，识别失败时需要显式配置。 |
| 暂不支持 | 不纳入本地 MVP；未来可在真实需求与测试覆盖后评估。 |
| 明确拒绝支持 | 因安全边界或产品范围，当前设计不接受。 |

## 运行时与包管理

| 维度 | 等级 | 第一版边界 |
| --- | --- | --- |
| Node.js 20 LTS | 支持 | 以固定 Linux 镜像摘要运行；具体补丁版本由 RunnerProfile 固定。 |
| Node.js 22 LTS | 有条件支持 | 仅当项目、依赖和官方样例通过兼容性回归。 |
| Node.js 18 或更旧 | 暂不支持 | 已结束或接近结束维护的运行时不进入首版承诺。 |
| npm | 支持 | 必须有 `package-lock.json`，默认使用可复现安装。 |
| pnpm | 有条件支持 | 必须有 `pnpm-lock.yaml` 并固定 `packageManager`；版本由 Corepack/RunnerProfile 固定。 |
| Yarn Classic / Berry | 暂不支持 | 待获得真实项目样本后分别评估，不将二者视为同一兼容面。 |
| 缺失 lockfile | 明确拒绝支持 | 依赖不可复现；必须先补齐并确认。 |
| 单一、标准 `package.json` 脚本 | 支持 | 至少显式提供 install/build/test/start 中任务所需脚本和端口。 |
| 动态 shell 拼接、交互式脚本 | 暂不支持 | 需要非交互、可超时、可记录退出码的命令。 |

## 项目结构

| 维度 | 等级 | 第一版边界 |
| --- | --- | --- |
| 单体 Node.js Web 应用 | 支持 | 单仓库、单主要服务、Linux 容器可运行。 |
| 前后端同仓库 | 有条件支持 | 启动顺序、端口、健康检查和构建命令必须显式配置。 |
| 纯后端 HTTP API | 有条件支持 | 可用于 API-first 验证，但不满足 M3 浏览器流程时相应项为 `unverifiable`，不能假装全量通过。 |
| Monorepo | 暂不支持 | 工作区选择、缓存、服务依赖和多端口超出第一版稳定范围。 |
| Windows-only 项目 | 明确拒绝支持 | 本地 MVP 执行面限定 Linux 临时容器。 |
| 通用桌面应用 | 明确拒绝支持 | 不属于本地 MVP。 |

## 数据与外部服务

| 维度 | 等级 | 第一版边界 |
| --- | --- | --- |
| SQLite | 支持 | 数据文件位于临时工作区；支持只读查询和前后状态对比。 |
| PostgreSQL | 暂不支持 | 后续需要受控服务编排、凭据注入和清理策略。 |
| Redis | 暂不支持 | 后续评估隔离实例、清理和确定性。 |
| Docker Compose | 暂不支持 | 本地 MVP 首版只承诺单目标容器与明确的受控依赖。 |
| 需要公共外部服务 | 暂不支持 | 网络默认受限；必须先有可替代的本地依赖或明确白名单策略。 |
| 需要生产/私人外部服务 | 明确拒绝支持 | 不允许把真实生产凭据或私人数据交给被验收环境。 |

## 配置与依赖

| 维度 | 等级 | 第一版边界 |
| --- | --- | --- |
| 非敏感环境变量 | 有条件支持 | 通过 RunnerProfile 显式白名单注入并记录键名；报告中不泄露敏感值。 |
| 真实 `.env` | 明确拒绝支持 | 不挂载、不复制宿主机真实 `.env`。 |
| 测试专用临时凭据 | 有条件支持 | 由可信控制面按任务注入、最小权限、运行后撤销并脱敏。 |
| 私有 npm 包 | 暂不支持 | Token、registry、缓存和审计策略尚未完成前不承诺。 |
| 纯 JavaScript/WASM 依赖 | 支持 | 仍需 lockfile 与固定镜像。 |
| 原生依赖 | 有条件支持 | 仅支持固定 Linux 镜像已具备构建前提且通过官方回归的依赖；否则为 `infrastructure_error` 或 `unverifiable`。 |
| 安装时需要特权、宿主设备或 Docker Socket | 明确拒绝支持 | 违反最小权限与隔离边界。 |

## 进入 M1 前必须明确的首批范围

M1 只接收：Node.js 20 LTS、npm lockfile、单体 Web 应用、单主要服务、标准非交互脚本、无私有包、无外部生产依赖、Linux 容器可运行、SQLite 可选。任何扩展必须先增加样例、威胁分析和量化门禁。

## M0-07 评审结论

M1 Runner 输入规范已固定在 [m1-runner-input-spec.md](m1-runner-input-spec.md)。本支持矩阵与该规范一致：M1 首版只承诺保守的 Node.js Web 项目执行面，重点是可复现安装、构建、测试、启动和健康检查，而不是支持任意工程形态。

M1 可以接受 npm 项目；pnpm 仅在存在 `pnpm-lock.yaml` 且 `package.json` 固定 `packageManager` 时有条件接受。Monorepo、Docker Compose、私有 registry、真实 `.env`、PostgreSQL、Redis、Windows-only 项目、桌面应用、未声明端口、多服务编排和外部生产服务均不进入首版承诺。

若项目无法提供 RunnerProfile 所需字段，或需要本矩阵明确拒绝的能力，Runner 应返回 `rejected_by_policy`、`unverifiable` 或 `infrastructure_error`，不得通过猜测命令、挂载宿主资源或放宽安全边界来“尝试跑通”。
